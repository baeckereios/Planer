2. Die Logik des Notlaufs: „Hey, ich laufe gerade reduziert!"

Damit das System erkennt, dass es im Notlauf operiert und bestimmte Produkte fehlen, brauchen wir eine Status-Validierung. Das lässt sich über den Datenbank-Manager lösen:
Situation System-Reaktion Meldung an den Nutzer
Produkt vorhanden (z.B. Schlawiner) Normale Berechnung, aber Warnung vor fehlendem Kontext. „Berechnung erfolgt auf Basis von Rohmengen (Notlauf). Wetter-Einfluss unbekannt."
Produkt fehlt (z.B. Spezialbrot) System prüft die letzte verfügbare Quelle im Goldstandard. „⚠️ Produkt im Notlauf-Modus nicht erfasst. Nutze historische Daten vom [Datum]."
Kontext fehlt (Wetter/Ferien) System schaltet auf einen „reinen Durchschnitt" ohne Gewichtung um. „Analyse eingeschränkt: Kontextfaktoren (Wetter/Ferien) fehlen für diesen Zeitraum."

3. Der Datenbank-Manager als „Weiche"

Der Datenbank-Manager muss lernen, zwei JSON-Dateien als eine logische Einheit zu betrachten. Die Logik für den Zugriff sieht in etwa so aus:
Datenquelle = { Goldstandard wenn vorhanden & Schalter = Gold; Notlauf wenn Schalter = Manuell; Historie (Gold) wenn Produkt im Notlauf fehlt }

4. Die Eingabemaske: Einfachheit als Schutz

Für die manuelle Datenbank brauchen wir eine Maske, die so radikal reduziert ist, dass sie keine Hürde darstellt:
  1. Datum & Wochentag: Werden automatisch gesetzt.
  2. Top-Produkte: Nur die Felder für die Mengen (z.B. Hasenberger, Schlawiner, Kornknacker).
  3. Der Poka Yoke Sicherheits-Check: Auch im Notlauf bleibt der Schutzmechanismus aus V109 aktiv. Die Buttons „❄️ FROSTER FERTIG?" und „❌ NOCH NICHT" müssen bedient werden, bevor der Datensatz gespeichert werden kann.

Der Goldstandard bleibt dadurch eine „saubere" Referenz, während die Notlauf-Datenbank dafür sorgt, dass die Mengenhistorie keine hässlichen Löcher bekommt.

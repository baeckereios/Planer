UI-Dateien (HTML)
index.html
UI · Einstieg
Cockpit/Startseite. Navigation zu allen Modulen. Theme-Toggle, Sprachwechsel.
setup.html
UI · 7-Schritt-Assistent
Sitzung aufbauen: Station → Produkte → Inventar (24h-Check) → Froster → Woche → Fehlmengen → Ziel. Schreibt BOS_SESSION in localStorage.
planer.html
UI · Kernmodul
Produktionsplanung. Liest BOS_SESSION. Zeigt Bedarfs-Ketten mit Bubbles (Bestand/Bedarf), Badges, Fortschrittsbalken. Nutzt BOS_BRAIN.
schnellrechner.html
UI · Sandbox
Einzelprodukt schnell durchrechnen ohne Setup-Flow. Liest BOS_INVENTUR direkt.
verbrauchsuebersicht.html
UI · Referenz
Tagesbedarfe aller Produkte als Tabelle. Reine Lese-Ansicht der Stammdaten.
frosterliste.html
UI · Druckvorlage
A4-Vorlage zur manuellen Papierzählung im Froster. Vollständig entkoppelt — kein Datenaustausch mit dem Rest.
📦
Daten- & Logik-Dateien (JS)
stammdaten.js
Daten · Statisch
Produktstammdaten: Name, Tagesbedarfe (7 Werte), Grill-Zuschlag, Einheit, Station. Unveränderlich zur Laufzeit. Setzt window.BOS_STAMMDATEN.
inventurdaten.js
Daten · Semi-Dynamisch
Ist-Bestand je Produkt mit Timestamp (ts). Wird von der Frosterzählung beschrieben. Setzt window.BOS_INVENTUR. 24h-Prüfung in setup.html.
froster_gehirn.js
Logik · Kernberechnung
Die einzige Berechnungsinstanz. BOS_BRAIN.calculateChain() berechnet die gesamte Bestandskette über den Planungszeitraum. Kennt Hamster, Grill, Froster-Status.
export.js
Logik · Output
Druckfenster (A4/80mm) und WhatsApp-Sharing. Setzt window.BOSExport. Wird nur von planer.html genutzt.
translations.js
Daten · i18n
Texte auf Deutsch, Englisch, Vietnamesisch. Nur in index.html und schnellrechner.html aktiv. Setup/Planer sind hart deutsch.
systemdesign.css
Design · Global
CSS-Variablen, Typografie, Dark/Light-Theme via data-theme. Gemeinsame Basis für alle HTML-Dateien.

window.BOS_BRAIN

Quelle: froster_gehirn.js V119 · Die einzige Berechnung im System

BOS_BRAIN.calculateChain(prodId, session, plannedProd)
// prodId:      "p1" (Key aus BOS_STAMMDATEN)
// session:     BOS_SESSION-Objekt
// plannedProd: Array mit geplanten Blechen pro Schritt [0,5,3,...]

// Rückgabe:
{
  productId, productName,
  isOk:       true/false,   // Kein negativer Bestand?
  maxDeficit: -7,           // Schlimmste Lücke (negativ)
  chain: [
    {
      stepIdx:           0,
      dayIdx:            6,       // JS-Wochentag (0=So..6=Sa)
      dayName:           "Samstag",
      need:              30,      // Angepasster Bedarf (nach Hamster/Grill)
      actualConsumption: 0,       // 0 wenn frosterDone && i < startStep
      planned:           5,       // Aus plannedProd[i]
      stockAfterNeed:    -7,      // Bestand nach Verbrauch (neg = Lücke)
      restAfter:         -2,      // Übergabe an nächsten Tag
      isWarning:         true
    }, ...
  ]
}

// Hamster-Multiplikatoren (session.weekConfig[dayIdx].hamster):
//   0 → ×1.0  (Normal)
//   1 → ×1.5  (🐹 Math.ceil)
//   2 → ×2.0  (🐹🐹 Ultra, Math.ceil)
// Grill: cfg.grill && p.sun > 0 → need += p.sun
// startStep: frosterDone ? 2 : 1

Bekannte Architektur-Grenzen — nicht als Kritik, sondern als ehrliche Bestandsaufnahme für Weiterentwicklungen.
⚠️
Architektur-Grenzen
  • 🗄️
    Kein Server, keine echte Datenbank. Alle Daten leben in statischen JS-Dateien oder localStorage. Inventurdaten müssen manuell in inventurdaten.js gepflegt werden — kein automatischer Rückschrieb aus dem Browser.
  • 🔄
    localStorage als einzige Übergabe. BOS_SESSION enthält nur die aktuelle Sitzung. Kein Verlauf, kein Rückblick, kein Vergleich mit gestern.
  • 🌐
    Mehrsprachigkeit nur teilweise. index.html und schnellrechner.html sind i18n-fähig (DE/EN/VI). Setup und Planer sind hard-coded Deutsch.
  • 🏭
    Nur Brötchenstraße & Nachtschicht produktiv. Frühschicht, Rondo, Konditorei, Versand haben keine Produkte in stammdaten.js hinterlegt.
  • 🖨️
    frosterliste.html ist vollständig entkoppelt. Keine Verbindung zum Rest — kein automatisches Befüllen aus BOS_INVENTUR, kein Zurückschreiben.
  • 📱
    Kein Offline-Caching (Service Worker). Funktioniert offline wenn alle Dateien gecacht sind, aber kein explizites PWA-Manifest.
Konzept für einen Experten-Modus — für erfahrene Mitarbeiter die den 7-Schritt-Assistenten nicht brauchen.
👤
Wer ist der Experte?

Jemand der die Produkte, Bedarfe und Bestände auswendig kennt. Er weiß was er tut, braucht keine Erklärungen und will direkt einsteigen. Der Assistent-Flow ist für ihn Overhead. Aber: er will dieselbe Berechnungsqualität.

Zwei Einstiegspunkte

📋 Direkt-Inventur Neue Datei

inventur_expert.html

  • Alle Produkte sofort sichtbar, gruppiert nach Station
  • Bestand direkt eingeben — ein Tap, eine Zahl, fertig
  • Timestamp automatisch gesetzt beim Speichern
  • Speichert in localStorage → setup.html 24h-Check greift sofort
  • Kein Assistent, kein Weiter-Button, keine Erklärungen
  • Optionaler Export als aktualisierte inventurdaten.js (Download)
⚡ Direkt-Planer Erweiterung

planer.html?expert=1

  • Kompakte Inline-Editoren oben: Bestand, Froster-Status, Wochenkonfig
  • Kein Setup-Durchlauf nötig — alles in einer Ansicht
  • Erfahrener Nutzer ändert Bestand direkt im Planer
  • Gleiche BOS_BRAIN-Berechnung, nur andere Oberfläche
  • Gelöst über URL-Parameter, kein separates Login
🚫
Was der Experten-Modus nicht ist
  • Kein verstecktes Power-User-Menü mit Schaltern
  • Kein separates Login oder Passwort
  • Kein "Profi-Theme" mit anderen Farben
  • Keine anderen Berechnungen — BOS_BRAIN bleibt die einzige Wahrheit
🗺️
Empfohlene Umsetzungsreihenfolge
  1. inventur_expert.html bauen — direktes Inventur-Eingabeformular mit localStorage-Persistenz
  2. In index.html als zweiten Einstieg unter "Arbeitsgeräte" aufnehmen
  3. URL-Parameter ?expert=1 in planer.html einbauen — zeigt Inline-Editoren oben
  4. Setup-Assistent bleibt unverändert für neue/wechselnde Mitarbeiter
